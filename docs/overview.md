# Overview

## The problem

Working with MongoDB day to day means constantly switching between the `mongosh` shell,
ad-hoc scripts, and heavyweight commercial GUIs to do simple things: look at what is
actually in a collection, fix one bad document, understand the shape of unfamiliar data,
or move a collection between environments. The shell is powerful but unfriendly for
browsing and editing; the full-featured commercial tools are large, license-gated, and
oriented toward database administration rather than everyday exploration.

MongoPal fills the middle ground: a fast, native desktop application for the
**explore → query → edit → move data** loop, with the safety rails (secure credential
storage, dry-run imports, confirmation dialogs) that make it usable against real
databases.

## Who it is for

Developers and data-focused engineers who read from and write to MongoDB regularly and
want a keyboard-driven GUI that starts instantly, keeps credentials safe, and does not
get in the way. It assumes familiarity with MongoDB concepts (databases, collections,
documents, BSON types) — it is a tool for people who already know MongoDB, not a
teaching aid.

## What it is (and isn't)

**What it is.** A single-user desktop client for one MongoDB deployment at a time
(standalone, replica set, or cluster reached through a single connection string). It
covers browsing collections, editing documents, running queries in several dialects
(native filter, mongosh, SQL, or natural language), inferring schema, managing indexes,
and importing/exporting data. It stores many saved connections and organizes them, but
it operates against one at a time.

**What it isn't.** MongoPal is deliberately not a database administration console: it
does not manage users and roles, configure replica sets or sharding, schedule backups,
or monitor cluster health beyond basic collection statistics. It is not a collaborative
or multi-user platform — there is no shared state, permissions model, or audit log. It
is not an analytics or dashboarding product, and it is not a visual aggregation-pipeline
builder — complex pipelines are expressed through mongosh scripts rather than a drag-and-drop
composer. Its writing surface is intentionally narrow: SQL query mode is read-only, and
document editing is one document at a time rather than batch transformation.

## Honest boundaries

MongoPal makes one meaningful guarantee — **credential confidentiality** — and it is
worth being precise about where that holds.

**Enforced.** Connection secrets (MongoDB passwords, SSH keys, TLS certs, proxy
credentials) are encrypted at rest with AES-256-GCM, and the encryption keys live in the
OS keyring (macOS Keychain, Windows Credential Manager, Linux Secret Service), not on
disk. The Anthropic API key for the AI assistant is stored keyring-only, with no file
fallback at all. Credentials are passed to child processes (`mongosh`, `mongodump`) via
environment variables rather than command-line arguments, so they do not appear in
process listings, and they are masked out of error messages. The AI assistant sends only
inferred schema (field names and types) to Anthropic — never document values. See
[decisions.md](decisions.md) for the mechanisms.

**Not a hard boundary.** On systems without a working keyring, encryption falls back to
an ephemeral key: data is still encrypted, but the encrypted connection files are no
longer portable across restarts. MongoPal is a local desktop app with full read/write
access to whatever the connection credentials permit — it enforces no additional
authorization layer of its own. It protects secrets at rest and in transit to
subprocesses; it does not sandbox what a connected user can do to the database.
