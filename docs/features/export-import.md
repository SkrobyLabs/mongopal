# Export & import

## What it does

Moves data in and out of MongoDB at collection, selected-database, or whole-database
scope. Exports can be written as JSON, CSV, or BSON archives (the latter via `mongodump`),
and export can be narrowed to specific databases or collections rather than the whole
instance. Imports accept JSON, CSV, BSON archives (via `mongorestore`), and MongoPal's own
exported bundles, with automatic file-format detection. Long-running transfers report live
progress with an ETA and can be paused, resumed, or cancelled. Before committing, an import
can be dry-run for a preview and conflict-resolution strategy for duplicate keys, and BSON
archives can be inspected — database/collection structure and document counts — without
importing.

## Why it exists

Everyday data work involves backups, migrations between environments, sharing a snapshot
with a teammate, and interchange with other tools. BSON via the official
`mongodump`/`mongorestore` gives true round-trip fidelity (indexes, types, sharding-aware
dumps), while JSON and CSV cover human-readable and spreadsheet interchange. Previewing an
archive and dry-running an import let a user verify contents before touching a live target.

## Constraints & trade-offs

- BSON export and import require `mongodump`/`mongorestore` to be installed locally;
  MongoPal detects their availability and points the user at the download otherwise. JSON
  and CSV paths use the Go driver directly and have no such dependency.
  See [decision 5](../decisions.md).
- CSV export flattens nested structures, emitting complex sub-objects as JSON strings.
