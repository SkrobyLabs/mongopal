# Query modes

## What it does

Lets users query a collection in whichever dialect fits them, all against the same data:

- **Native MongoDB filter** — a BSON filter object with syntax highlighting, validation,
  and schema-aware autocomplete.
- **Aggregation pipelines** — full pipelines executed through the driver, with results and
  execution time returned to the UI.
- **mongosh scripts** — arbitrary JavaScript shell scripts executed through the local
  `mongosh` binary, with or without a database context. This doubles as a general script
  runner for migrations, initialization, and diagnostics — not just read queries.
- **SQL mode** — a read-only `SELECT` subset (WHERE, ORDER BY, GROUP BY, HAVING, LIMIT)
  converted to a MongoDB query on the fly, with a live preview of the generated query and
  schema-aware autocomplete.

Query history is recorded automatically, and frequently-used queries can be kept as named
[saved queries](../glossary.md). For generating a query from a plain-language description,
see the separate [AI query assistant](ai-query-assistant.md).

## Why it exists

Different users reach for different tools: shell veterans want mongosh, analysts think in
SQL, and pipeline work needs raw aggregation. Offering all of them against one collection
removes the friction of translating between mental models — and keeping the full mongosh
escape hatch means anything the higher-level modes can't express is still reachable.

## Constraints & trade-offs

- SQL mode is deliberately read-only and closed-grammar: no INSERT/UPDATE/DELETE, no JOIN,
  subqueries, or CTEs. Unsupported constructs fail with a hint pointing at mongosh or the
  document editor. The converter runs entirely client-side for instant preview
  (see [decision 6](../decisions.md)).
- mongosh script mode depends on the `mongosh` binary being installed; MongoPal detects its
  availability and reports when it is missing.

Code entry point: `internal/script/`, `frontend/src/utils/sqlConverter/`,
`internal/document/crud.go` (aggregation)
