# Diagnostics & monitoring

## What it does

Brings the operational picture into the same tool as the data:

- **Indexes** — list, create, and drop indexes on a collection.
- **Explain** — run a query's execution plan and read the winning stage, index usage, and
  whether it fell back to a collection scan.
- **Collection statistics** — document count, size, and storage.
- **Server info** — a diagnostics modal covering server version, topology and replica-set
  members, host resources, uptime, connection and operation counters, and network I/O,
  with a searchable raw-JSON tree for anything not surfaced directly.
- **Performance metrics** — MongoPal's own Go runtime health: memory, goroutines, GC
  activity, connection-pool utilization, and uptime, with a manual GC trigger.
- **Debug logging** — a toggleable, categorized log stream (frontend and backend sources)
  that can be exported to JSON for troubleshooting.

## Why it exists

Exploration naturally turns into "why is this query slow?" or "is the server healthy?"
MongoPal keeps the answers in the same window: inspect indexes and the explain plan, check
server and collection stats, and — when something misbehaves — capture a debug log without
leaving for a separate admin console or the shell.

## Constraints & trade-offs

- This is diagnostic tooling, not database administration — it stops short of user/role
  management, replica-set or shard configuration, and cluster health alerting (see the
  [overview non-goals](../overview.md)).
- Explain and server info are presented close to the driver's/server's raw form rather than
  as visualized plans or dashboards.
- Performance metrics describe the MongoPal client process, not the MongoDB server.

Code entry point: `internal/database/` (`indexes.go`, `explain.go`, `stats.go`),
`internal/connection/` (server info), `internal/performance/`, `internal/debug/`,
