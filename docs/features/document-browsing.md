# Document browsing & editing

## What it does

Browses a collection's documents in either a table view or a raw JSON view, paginated to
keep large collections responsive. Documents are edited in an embedded Monaco editor with
syntax highlighting, formatting, and JSON validation before save; new documents can be
inserted. Two documents can be compared side by side with a visual diff. Multiple documents
can be selected and acted on in bulk — deleted or exported — and whole collections can be
cleared or dropped behind confirmation. Table columns size themselves from field names and
types, and their widths persist per collection. Documents and field values can be copied to
the clipboard as Extended JSON.

## Why it exists

The core reason a GUI beats the shell for everyday work: seeing what is actually in a
collection at a glance, and fixing an individual document without hand-writing an update
query. The table view surfaces data intuitively; the JSON editor and diff view handle the
awkward BSON types and let you audit exactly what changed before committing it.

## Constraints & trade-offs

- Documents are shown and edited as MongoDB Extended JSON (e.g. `{"$oid": "..."}`,
  `{"$date": "..."}`) so that every fetch → edit → save round-trip preserves exact BSON
  types — more verbose than shell shorthand, but lossless. See [decision 4](../decisions.md).
- Editing is one document at a time; bulk actions are limited to delete and export, not
  find-and-modify transforms.
- Destructive actions (delete, clear, drop) are gated by confirmation and, for connections
  configured that way, a deliberate delay. See [Connection management](connection-management.md).
- Browsing is page-based rather than an infinite cursor.
